# 根据古诗文文档，构建知识库的方案

## 0. 背景与目标

今天（2026-07-21）的任务是构造一个知识库，内容是浙江省中考语文大纲要求默写的古诗文。要求：

- 按 **年级 + 学期** 分类
- 支持后续 **上下句填空 + 理解性默写** 两种题型的出题数据需求
- 标注 **易错字**（按需：仅当题目答案存在易错字、易混淆字、生僻字时才标注，无则不标）
- 最终产出一份 md 文档，后续需要把这份 md 文档做成知识库
- 知识库要能根据 **年级 + 学期** 进行检索（题型、数量为出题层参数，不由知识库检索层负责）

本文档只讲"理论方案"——即：拿到一份按上述规范整理好的古诗文 md 文档后，如何把它做成一个可按年级+学期检索的知识库。具体的数据整理（篇目录入、易错字标注）属于后续工作。

> 与已有技术设计文档的关系：`technology_documents/classical-poetry-dictation/technical-design.md` 已经把知识库方案定为"结构化数据（非 RAG）"，并把题型（上下句填空/理解性默写/混合）、数量（默认6可指定）作为出题层参数。本文档是这一结论的**理论铺垫**，解释为什么这么选、md 文档在其中扮演什么角色、检索是怎么实现的。

---

## 1. 先搞清楚：LLM 应用里的"知识库"是什么

在 LLM 应用语境下，"知识库"不是数据库课本里那个抽象概念，而是指**给大模型外挂一份它训练时没见过、或记不准的私有数据，让模型回答时只基于这份数据，而不是自由发挥**。

对本项目而言，古诗文默写出题的准确性命脉就在于：题目必须 100% 来自浙江省中考提纲，一字不差。LLM 自己"背"古诗文会出现错字、漏字、张冠李戴，所以必须用知识库把它**约束**住。

知识库的构建方式，业界主要有三种范式：

| 范式 | 核心机制 | 适用场景 | 典型实现 |
|------|---------|---------|---------|
| **结构化数据** | 把数据存成有字段的数据表/对象，用字段精确匹配 | 数据有清晰的字段结构，查询条件是精确的（如年级=八年级） | SQLite / JSON / MySQL |
| **向量检索 / RAG** | 把文本切成块，用 embedding 向量化，查询时按语义相似度召回 | 数据是非结构化文本，查询是模糊语义（如"讲友情的诗"） | Chroma / FAISS / Milvus |
| **混合检索** | 先用结构化字段过滤，再用向量重排 | 既要精确过滤，又要语义排序 | Metadata filter + vector |

**关键认知**：选哪种范式，不取决于"哪种更高级"，而取决于**数据形态**和**查询方式**。下面结合本项目的实际情况分析。

---

## 2. 为什么古诗文场景选"结构化数据"而不是 RAG

这是本文档最核心的理论判断，逐条论证：

### 2.1 数据形态：高度结构化

浙江省中考提纲的古诗文，每篇都有清晰、确定、可枚举的属性：

| 属性 | 是否结构化 | 取值示例 | 是否必需 |
|------|-----------|---------|---------|
| 篇目名称 | 是 | 岳阳楼记 | 必需 |
| 作者 | 是 | 范仲淹 | 必需 |
| 年级 | 是（枚举） | 七年级 / 八年级 / 九年级 | 必需 |
| 学期 | 是（枚举） | 上册 / 下册 | 必需 |
| 全文 | 半结构化 | 一段标准文本 | 必需 |
| 体裁 | 是（枚举） | 诗词 / 文章 | 可选（内部统计使用，不作为筛选维度） |
| 易错字 | 结构化列表 | [{char, confused_with, hint}] | 可选（按需标注，无则空） |
| 理解性默写提示 | 文本列表 | ["表达旷达胸襟", ...] | 可选（辅助 LLM 生成理解性默写题干） |

几乎所有用于"检索/筛选"的字段都是枚举值或精确值，这正是结构化数据库最擅长的场景。

> **注**：根据 PRD 2.2.2，教师可筛选维度仅保留「年级 + 学期」。体裁虽在知识库中保留为字段，但仅用于 Agent 内部统计，不暴露为筛选条件。题型（上下句填空/理解性默写/混合）与数量（默认6可指定）是**出题层参数**，不属于知识库检索维度。

### 2.2 查询方式：精确匹配，不是语义匹配

教师的查询是"八年级下册""七年级上册""整个三年"这类**精确条件**，不是"找一首讲思乡的诗"这种**语义查询**。

- "八年级下册" → `WHERE grade='八年级' AND semester='下册'`，精确命中，不存在"接近八年级"这种模糊语义。
- 向量检索的核心价值是处理"语义相似但字面不同"的查询，而本场景根本没有这种需求。

强行用 RAG，相当于用语义相似度去解一个本该用 `WHERE` 子句解决的问题，不仅没有收益，还会引入"召回不准"的风险。

### 2.3 数据规模：小

浙江省中考提纲的古诗文约 60-80 篇，全量加载到内存都毫无压力。向量检索是为大规模非结构化语料设计的，几十条数据用向量库是杀鸡用牛刀。

### 2.4 准确性要求：100% 可控

默写出题要求"一字不差"。结构化数据的字段是人工录入的确定值，可审计、可校对；而 RAG 的召回结果带有概率性，无法保证每次都返回同一批篇目，不利于出题的稳定性与可复现性。

### 2.5 结论

> 古诗文知识库采用**结构化数据方案**：md 文档作为人类可读的数据源，解析成结构化数据后，按字段精确检索。不引入向量检索。

这条结论与已有技术设计文档一致，本文档为其提供理论依据。

> 补充：知识库检索层只负责"按年级+学期拿出篇目"；**题型（上下句填空/理解性默写/混合）与数量（默认6可指定）属于出题层参数**，由 `select_questions` Tool 处理，不在知识库检索职责内。理解性默写的语义题干由 LLM 即时生成（可由 `comprehension_hints` 辅助），不预存于知识库。

## 3. md 文档在整个方案中的定位

理解了"用结构化数据"之后，一个自然的问题是：**既然要结构化数据，为什么还要先写 md 文档？直接写 JSON/SQLite 不行吗？**

### 3.1 md 文档的角色：人类可读、可维护的数据源

直接维护 JSON 对教师极不友好--几百行 JSON 里改一个字要小心翼翼找括号、引号、逗号，容易出错，也难以通读全文。

md 文档的优势在于：

- **教师可读、可编辑**：教师可以用熟悉的 Markdown 语法直接增删篇目、调整易错字标注，无需懂编程。
- **可被规则解析**：只要 md 文档遵循固定格式约定，程序就能稳定地把它解析成结构化数据。
- **可版本管理**：md 是纯文本，天然适合 Git diff，篇目调整都有迹可循。
- **单一数据源（Single Source of Truth）**：教师只维护这一份 md，程序负责解析转换，避免"数据散落在多处、彼此不一致"。

### 3.2 数据流向

```
教师维护          程序解析            程序检索          Agent 使用
  │                 │                   │                 │
  ▼                 ▼                   ▼                 ▼
[md 文档]  -->  [结构化数据]  -->  [按年级+学期过滤]  -->  [出题/查询]
(人类可读)     (JSON/SQLite)      (精确字段匹配)        (Tools 调用)
```

md 文档是**起点**，不是**终点**；知识库的真正载体是解析后的结构化数据。这一层"解析"是本方案的关键工程环节。

## 4. md 文档的规范结构设计

要让程序能稳定解析，md 文档本身必须有**固定、可预期的结构约定**。本节定义这份 md 文档应该长什么样。

### 4.1 结构设计原则

1. **层级即分类**：用 Markdown 标题层级天然表达"年级 > 学期 > 篇目"的从属关系，这样既人类可读，又能被解析器按标题层级提取。
2. **篇目元信息用键值对**：每篇用统一的"字段: 值"格式列出作者、体裁、易错字等，解析器用正则即可稳定提取。
3. **全文用代码块包裹**：古诗文全文放进代码块，避免与 Markdown 语法冲突（如标点、特殊符号），也便于解析器整块提取。
4. **字段命名稳定**：一旦约定字段名，全文档统一，不混用同义词。

### 4.2 推荐的 md 文档结构

````markdown
# 浙江省中考语文默写古诗文知识库

> 说明：本知识库按 年级 > 学期 > 篇目 三级组织。
> 易错字按需标注，无易错字则不写"易错字"行。
> 理解性默写提示按需标注，用于辅助 LLM 生成理解性默写题干。

## 七年级

### 上册

#### 《观沧海》曹操
- 体裁：诗词
- 易错字：碣石（易错为"竭石"）、澹澹（易错为"淡淡"）
- 理解性默写提示：描写大海宏伟景象的诗句
- 备注：建安文学代表作

```
东临碣石，以观沧海。
水何澹澹，山岛竦峙。
...
```

#### 《闻王昌龄左迁龙标遥有此寄》李白
- 体裁：诗词

```
杨花落尽子规啼，闻道龙标过五溪。
我寄愁心与明月，随君直到夜郎西。
```

### 下册

#### 《木兰诗》佚名
- 体裁：诗词

```
唧唧复唧唧，木兰当户织。
...
```

## 八年级

### 上册
...

### 下册
...

## 九年级

### 上册
...

### 下册
...
````

### 4.3 字段约定

| 字段 | 是否必需 | 取值 | 说明 |
|------|---------|------|------|
| 体裁 | 可选 | 诗词 / 文章 | 内部统计使用，不作为筛选维度 |
| 易错字 | 可选 | 自由文本（建议格式：字（易错为"某"），多项用顿号分隔） | 按需标注，仅当该篇存在易错字时才写；无则不写此行 |
| 理解性默写提示 | 可选 | 自由文本 | 辅助 LLM 生成理解性默写题干；无则由 LLM 自行据文意生成 |
| 备注 | 可选 | 自由文本 | 教师备注，解析时可选提取 |

> 注1：易错字遵循 PRD 2.2.1 的"按需标注"原则--仅当题目答案中存在易错字、易混淆字、生僻字时才标注，无则不标，避免噪音。是否为易错字由 AI 自动判别。
> 注2：理解性默写提示是**辅助字段**，知识库不预存题干本身；题干由 LLM 即时生成，保证多样性。

## 5. 解析方案：从 md 到结构化数据

这一节讲程序如何把 md 文档解析成可检索的结构化数据。

### 5.1 解析的核心任务

解析器需要从 md 中提取出以下结构化对象（一篇一个）：

```
{
  "id": "七上_观沧海",           // 由年级学期+篇目生成，保证唯一
  "title": "观沧海",
  "author": "曹操",
  "grade": "七年级",
  "semester": "上册",
  "genre": "诗词",               // [内部字段] 可选，缺失则置 null
  "error_prone_chars": [        // [可选] 易错字列表，按需标注；无则空数组 []
    {"char": "碣", "confused_with": "竭", "hint": "\"碣\"指石碑，\"竭\"指尽"}
  ],
  "comprehension_hints": [      // [可选] 理解性默写提示，辅助 LLM 生成题干；无则空数组 []
    "描写大海宏伟景象的诗句"
  ],
  "remark": "建安文学代表作",
  "content": "东临碣石，以观沧海。\n水何澹澹，山岛竦峙。\n..."
}
```

其中 `grade` 和 `semester` 是检索的关键字段，它们**不出现在篇目内部**，而是由篇目所在的标题层级（`## 年级` / `### 学期`）决定。这是解析器要处理的核心逻辑。

`genre`、`error_prone_chars`、`comprehension_hints` 均为可选字段，缺失时分别置 null / 空数组，不影响检索；其中 `genre` 仅用于 Agent 内部统计，`error_prone_chars` 供易错字标注 Tool 使用，`comprehension_hints` 供理解性默写题干生成使用。

### 5.2 解析算法思路（基于标题层级状态机）

由于 md 用标题层级表达分类，解析器本质是一个**"状态机"**：遍历每一行，根据标题级别更新当前所属的年级/学期上下文，遇到篇目标题（`####`）时，把当前上下文作为该篇目的 `grade`/`semester`。

伪代码：

```python
def parse_md(md_text):
    poems = []
    current_grade = None
    current_semester = None
    current_poem = None          # 正在收集的篇目对象

    for line in md_text.splitlines():
        if line.startswith("## "):
            # 二级标题：年级，如 "## 七年级"
            current_grade = line[3:].strip()
        elif line.startswith("### "):
            # 三级标题：学期，如 "### 上册"
            current_semester = line[4:].strip()
        elif line.startswith("#### "):
            # 四级标题：篇目，如 "#### 《观沧海》曹操"
            if current_poem:
                poems.append(current_poem)
            title, author = parse_title(line)   # 解析出篇名和作者
            current_poem = {
                "title": title,
                "author": author,
                "grade": current_grade,
                "semester": current_semester,
                "genre": None,                      # 可选字段，默认 null
                "error_prone_chars": [],            # 可选字段，默认空数组（按需标注）
                "comprehension_hints": [],          # 可选字段，默认空数组
            }
        elif line.startswith("- ") and current_poem:
            # 列表项：键值对元信息，如 "- 体裁：诗词"
            key, value = parse_kv(line)
            if key in ("体裁", "备注"):
                current_poem[normalize_key(key)] = value
            elif key in ("易错字", "理解性默写提示"):
                # 多值字段：解析为列表，按顿号或分号拆分
                current_poem[normalize_key(key)] = split_values(value)
        elif line.startswith("```") and current_poem:
            # 代码块：全文
            current_poem["content"] = read_code_block(...)
        # 其他行忽略

    if current_poem:
        poems.append(current_poem)
    return poems
```

**要点**：

1. **状态由标题层级维护**：解析器始终记住"当前在哪一年级、哪一学期"，篇目自动继承这个上下文。这样教师只需把篇目放到对应章节下，无需在每个篇目里重复写年级学期。
2. **字段提取用键值对正则**：`- 字段：值` 格式稳定，`^-\s*(.+?)[:：]\s*(.+)$` 即可匹配。
3. **全文用代码块整块提取**：遇到 ` ``` ` 开始记录，遇到下一个 ` ``` ` 结束。
4. **生成唯一 id**：用 `年级学期_篇名`（如 `七上_观沧海`）或序号，保证篇目可被唯一引用。
5. **可选字段缺失不报错**：`genre` 缺失置 null，`error_prone_chars`/`comprehension_hints` 缺失置空数组，确保后续出题逻辑可平滑降级。

### 5.3 解析的健壮性保障

实际 md 文档可能有不规范之处，解析器需做容错：

| 场景 | 容错策略 |
|------|---------|
| 篇目标题里作者缺省（只有 `#### 《观沧海》`） | author 置空，不报错 |
| 键值对用了英文冒号 `:` | 正则同时匹配中英文冒号 |
| 全文没用代码块包裹 | 回退策略：把篇目标题之后、下一个篇目标题之前的非元信息行当作全文 |
| 出现未知字段（如 `- 朝代：汉末`） | 一并收入篇目对象，不报错，便于扩展 |
| 同一年级学期下没有篇目 | 跳过，不产生空记录 |
| 易错字/理解性默写提示字段缺失 | 置空数组 `[]`，不报错 |

解析完成后应输出一份**校验报告**：总篇目数、按年级学期分布、必需字段缺失的篇目列表（如某篇没有 author 或 content），以及可选字段（易错字）的覆盖率统计，便于教师核对数据完整性。

## 6. 检索方案：按年级 + 学期检索的实现

知识库构建的最终目的是被检索使用。本节讲检索的具体实现。

### 6.1 检索的本质：字段过滤

由于数据是结构化的，"按年级+学期检索"本质上就是对篇目列表做**字段过滤**：

```
结果 = [p for p in 所有篇目 if 满足条件(p)]
```

支持的条件维度如下：

| 维度 | 字段 | 取值 | 是否对外暴露 | 用途 |
|------|------|------|-------------|------|
| 年级 | `grade` | 七年级/八年级/九年级/全部 | 是 | 教师筛选条件 |
| 学期 | `semester` | 上册/下册/全部 | 是 | 教师筛选条件 |
| 体裁 | `genre` | 诗词/文章 | 否（内部使用） | Agent 内部统计使用，不暴露给教师筛选 |

> **经与语文教师确认（2026-07-21）**：教师实际出题时仅按「年级 + 学期」筛选篇目，不会按体裁筛选。因此 `genre` 虽在知识库中保留为字段，但仅用于 Agent 内部统计，不作为教师可调用的筛选维度暴露。
>
> **题型与数量不属于检索维度**：题型（上下句填空/理解性默写/混合）与数量（默认6可指定）是**出题层参数**，由 `select_questions` Tool 接收并处理，知识库检索层只负责"按年级+学期拿出篇目"，不感知题型与数量。

### 6.2 检索接口设计

封装一个统一的检索函数，Agent 的 Tool 直接调用它：

```python
class PoetryKnowledgeBase:
    def __init__(self, poems):
        self.poems = poems   # 解析后的篇目列表

    def query(self, grade=None, semester=None):
        """
        按条件检索篇目（对外仅暴露年级+学期两个筛选维度）。
        任一参数为 None 或 "全部" 时表示不限制该维度。
        genre 不作为对外筛选参数，仅供 Agent 内部统计读取。
        """
        result = self.poems
        if grade and grade != "全部":
            result = [p for p in result if p["grade"] == grade]
        if semester and semester != "全部":
            result = [p for p in result if p["semester"] == semester]
        return result

    def get_by_id(self, poem_id):
        """按 id 取单篇，供易错字、理解性默写提示等 Tool 使用"""
        return next((p for p in self.poems if p["id"] == poem_id), None)
```

### 6.3 典型检索场景示例

| 教师需求 | 检索调用 | 返回 |
|---------|---------|------|
| "八年级下册" | `query(grade="八年级", semester="下册")` | 八年级下册全部篇目 |
| "七年级"（整年） | `query(grade="七年级")` | 七年级上下册全部篇目 |
| "整个三年" | `query()` | 全部篇目 |
| "出10道八年级下册理解性默写" | `query(grade="八年级", semester="下册")` 后由 `select_questions(count=10, question_type="comprehension")` 处理 | 八下全部篇目（题型与数量由出题层处理） |
| "出一半理解性默写" | `query(...)` 后由 `select_questions(question_type="mixed")` 处理 | 检索层只返回篇目，混合题型分配在出题层 |

可以看到，"年级+学期"检索就是组合条件过滤，逻辑简单、结果确定、可复现。这正是结构化数据方案带来的好处--**没有概率性，每次检索同一条件必然返回同一结果**。

> 题型（含混合比例）与数量（如5/10/20道，默认6道）的解析与分配，由 Agent 在出题层完成，详见技术设计文档的 `select_questions` Tool 与出题策略 Skill。

### 6.4 检索结果如何被 Agent 使用

检索结果返回给 Agent 后，配合后续的出题逻辑完成出题。这一层在已有技术设计文档的 Tools/Skills 章节已有定义，本文档不展开。

出题层的关键衔接点：
- **题目抽取**：`select_questions` 从检索结果中按出题策略选出指定数量的题目（候选不足时放宽篇目多样性约束）
- **题型处理**：`question_type` 决定每题形态--上下句填空直接用 `prev_line`/`next_line` 生成；理解性默写由 LLM 基于 `comprehension_hints`（若有）或诗句语义即时生成题干
- **易错字标注**：对选中的题目调用易错字 Tool 读取 `error_prone_chars`（按需，无则不输出易错字行）

需要强调的是：**检索层只负责"按条件拿出篇目"，不负责"出题策略"**。职责单一，便于维护和测试。

## 7. 存储载体选择：JSON 还是 SQLite

解析得到的结构化数据，存到哪里？两个阶段的建议：

| 阶段 | 载体 | 理由 |
|------|------|------|
| 开发期 | JSON 文件 | 纯文本、可读、可 Git diff，调试方便；数据量小（60-80 篇），全量加载无压力 |
| 稳定期 | SQLite | 支持索引和 SQL，查询更规范；但需写迁移脚本，开发期不必引入 |

**关键点**：无论哪种载体，都应封装在一个 `PoetryKnowledgeBase` 类后面，上层（Agent/Tools）只调用 `query(...)` 等方法，不感知底层是 JSON 还是 SQLite。这样载体可平滑切换，不影响业务逻辑。

本期任务（整理 md 文档 + 构建知识库）建议先走 JSON 路线：md -> 解析 -> `poems.json` -> `PoetryKnowledgeBase` 加载使用。

## 8. 数据质量与维护

知识库的价值取决于数据质量。本节列出几个维护要点。

### 8.1 数据校验

解析后应校验以下规则，发现问题立即报告：

**必需字段校验（缺失即报错）**：
- 每篇必有：title、author、grade、semester、content
- grade 取值在 {七年级, 八年级, 九年级} 内
- semester 取值在 {上册, 下册} 内
- title 不重复（同一篇不应在不同年级学期重复出现，除非确有安排）
- content 非空

**可选字段校验（缺失仅告警/统计，不阻断）**：
- genre 取值若存在，应在 {诗词, 文章} 内；缺失则置 null
- error_prone_chars、comprehension_hints 若存在应为列表；缺失则置空数组
- 统计可选字段覆盖率（如"易错字标注率 60%"），供教师评估数据完整度

### 8.2 单一数据源原则

**只维护 md 文档这一份数据源**。`poems.json` 等结构化数据是**生成产物**，由解析器从 md 生成，不要手工编辑。这样能避免"md 和 JSON 不一致"的经典问题。

建议在 `.gitignore` 中考虑是否忽略生成产物，或在生成产物头部加注释标明"本文件由 md 自动生成，请勿手工编辑"。

### 8.3 数据更新流程

教师增删/修改篇目时的标准流程：

```
教师编辑 md 文档  ->  运行解析器  ->  生成新的 poems.json  ->  运行校验  ->  知识库更新完成
```

这个流程可以做成一个脚本（如 `python build_kb.py`），一条命令完成"解析 + 校验 + 生成"，降低维护成本。

## 11. 关键结论速览

1. **知识库范式选"结构化数据"，不选 RAG**。原因是数据结构化程度高、查询是精确匹配、数据规模小、准确性要求 100% 可控。
2. **md 文档是人类可读的数据源，不是知识库本身**。知识库的真正载体是解析后的结构化数据。
3. **md 文档用标题层级表达"年级>学期>篇目"分类**，篇目用键值对标注元信息，全文用代码块包裹，保证可被规则稳定解析。
4. **解析器是"标题层级状态机"**，靠维护"当前年级/学期"上下文，给篇目自动打上分类标签；可选字段缺失不报错，平滑降级。
5. **检索就是按字段精确过滤**，`grade + semester` 是唯一对外检索维度，体裁仅内部使用不暴露。
6. **题型与数量是出题层参数，不属于知识库检索**：题型（上下句填空/理解性默写/混合）与数量（默认6可指定）由 `select_questions` Tool 处理；理解性默写题干由 LLM 即时生成（可由 `comprehension_hints` 辅助）。
7. **易错字按需标注**：仅当题目答案存在易错字时才标注，无则不输出，避免噪音；是否为易错字由 AI 自动判别。
8. **单一数据源原则**：只维护 md，结构化数据由解析自动生成，避免数据不一致。
9. **存储先 JSON 后 SQLite**，通过 `PoetryKnowledgeBase` 类封装，载体可平滑切换。
